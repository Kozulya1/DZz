# Домашнее задание к занятию «Репликация и масштабирование. Часть 2»
## Задание 1
### Активный master-сервер и пассивный репликационный slave-сервер
1.Повышение отказоустойчивости: В случае выхода из строя мастер-сервера, у вас есть его точная копия, на которую можно быстро переключить приложение. Это значительно сокращает время простоя по сравнению с восстановлением из резервной копии.

2. Резервное копирование без нагрузки на мастер: Вы можете выполнять резервное копирование на слейве, не создавая дополнительной нагрузки на основной рабочий сервер. Это гарантирует, что производительность мастер-сервера не пострадает во время создания бэкапа.

3. Распределение нагрузки для отчетов и аналитики: Тяжелые аналитические запросы, формирование отчетов или выгрузки данных можно выполнять на слейве. Это позволяет сохранить ресурсы мастер-сервера для обработки транзакций записи от пользователей.

### 2. Master-сервер и несколько slave-серверов

1. Значительное масштабирование для чтения: Нагрузка на чтение распределяется между несколькими слейвами. Это позволяет обслуживать гораздо большее количество пользователей, выполняющих операции чтения, без увеличения нагрузки на единственный мастер.

2. Геораспределение данных: Вы можете разместить несколько слейвов в разных географических регионах. Это значительно снижает задержки для пользователей, так как они читают данные с ближайшего к ним сервера.

3. Повышенная отказоустойчивость и гибкость: Наличие нескольких копий данных делает систему еще более надежной. Выход из строя одного из слейвов не влияет на работу приложения — есть другие доступные слейвы. Кроме того, один из слейвов всегда можно выделить для специфических задач, например, для долгих миграций данных или сложных аналитических запросов, не влияя на другие.

## Задание 2
### План вертикального шардинга 

Сервер A (Основной):	
#### пользователи -	user_id, name, email, registration_date, city, last_login	(Часто используемые данные для быстрых запросов (профили, списки)).
#### книги -	book_id, title, author, genre, year, price, store_id	(Основная информация о книге (без тяжелых описаний)).
#### магизины - store_id, name, address, city, phone, email	(Основные данные магазина (без служебных заметок)).

Сервер B (Расширение):
#### пользователи - users_private	user_id, password_hash, phone	(Чувствительные данные. Доступ строго ограничен).
#### книги - books_ext	book_id, description, full_text, table_of_contents	(Тяжелые текстовые поля. Используются только при детальном просмотре).
#### магазины - stores_ext	store_id, manager_notes, bank_details, legal_info	(Служебные и финансовые данные магазинов).

Режимы работы серверов при вертикальном шардинге
#### Сервер A - режим	Master (Read-Write), разрешены операции:	SELECT, INSERT, UPDATE, DELETE.	Принимает все запросы на запись. С него данные реплицируются на Сервер B.
#### Сервер B - режим Slave (Read-Only), разрешены операции SELECT.	Принимает запросы на чтение больших данных. Получает обновления с Сервера А через асинхронную репликацию.

### План горизонтального шардинга 
Разбиваем каждую таблицу по диапазонам первичных ключей.

Шард 1 хранит диапозоны user_id, book_id и store_id с 1 по 1 000 000
Шард 2 хранит диапозоны user_id, book_id и store_id с 1 000 001 по 2 000 000
Шард 3 хранит диапозоны user_id, book_id и store_id с 2 000 001 по 3 000 000

Режимы работы серверов при горизонтальном шардинге:
Каждый шард	Master (Read-Write)	SELECT, INSERT, UPDATE, DELETE.	Каждый шард — это полноценный мастер, обслуживающий свой диапазон ID.
